 STEP 5: NEW — Nonce Manager service (backend only)
───────────────────────────────────────

@workspace Create a NonceManagerService for the wallet-service backend. This handles transaction ordering to prevent nonce collisions when automating multiple sub-wallet transactions.

NonceManagerService (nonce-manager.service.ts):
Uses Redis for lock-free nonce tracking per wallet address.

1. acquireNonce(walletAddress: string):
   - Redis key: "nonce:{address}"
   - If key doesn't exist: fetch from chain via provider.getTransactionCount(address, 'pending')
   - Atomically increment and return
   - Set TTL lock: "nonce-lock:{address}:{nonce}" = true, expire 120s
   - Return { nonce, lockId }

2. confirmNonce(walletAddress, nonce):
   - Remove lock key
   - Update confirmed nonce counter

3. releaseNonce(walletAddress, nonce):
   - Called when tx was never submitted (error before send)
   - Remove lock, do NOT increment

4. detectStuck(walletAddress):
   - Scan lock keys older than 5 minutes
   - Return stuck nonce list
   - Log warning

5. replaceByFee(walletAddress, stuckNonce, newGasPrice):
   - Build replacement tx with same nonce, higher gas
   - Sign and submit
   - Return new txHash

6. resyncFromChain(walletAddress):
   - Fetch on-chain nonce via getTransactionCount(address, 'latest')
   - Reset Redis counter
   - Clear all locks for this address
   - Return { syncedNonce }

7. Redis data structure:
   - "nonce:current:{address}" — current nonce counter (string, INCR)
   - "nonce:lock:{address}:{nonce}" — lock with TTL (SET EX 120)
   - "nonce:pending:{address}" — sorted set of pending nonces with timestamps

Include a background job that runs every 60 seconds:
- For each active wallet, call detectStuck()
- If stuck nonces found: attempt replaceByFee with 1.3x gas
- If replaceByFee fails 3 times: call resyncFromChain()
- Log all actions

Write jest tests mocking Redis and ethers provider.

SwapService (swap.service.ts):

1. createSwapTask(params):
   a. Get wallet signer from WalletVault (by walletId, NOT raw key)
   b. For each target in params.targets:
      - Build Uniswap V3 exactInputSingle call:
        tokenIn = params.tokenIn address
        tokenOut = target.address
        fee = params.feeTier
        amountIn = calculated from target.amount and current price
        amountOutMinimum = target.amount * (1 - params.slippage/100)
        deadline = block.timestamp + 300
      - If params.chunkEnabled and amountIn > chunkSize:
        Split into chunks, queue each with delay
      - If params.useFlashbots:
        Submit via Flashbots RPC endpoint
      - Else:
        Submit via regular provider
   c. Track each swap as a transaction record
   d. Return taskId for monitoring

2. Uniswap integration:
   - Import Router ABI and Quoter ABI as constants
   - Router address: 0xE592427A0AEce92De3Edee1F18E0157C05861564 (V3)
   - Quoter address: 0xb27308f9F90D607463bb33eA1BeBb41C27CE5AB6 (V3)
   - Handle token approval before first swap per token
   - Use multicall for batch swaps when possible

3. Price oracle for the "≈ $USD" display in UI:
   GET /api/swap/quote { tokenIn, tokenOut, amount }
   - Call Uniswap Quoter.quoteExactInputSingle()
   - Fetch USD price from Coingecko or Chainlink
   - Return { amountOut, usdValue, priceImpact }

 STEP 7: NEW — Return wallet service with time-lock
───────────────────────────────────────

@workspace Create the return wallet service that powers the "Create Withdraw Task" modal.

When user submits the withdraw form, backend receives:
{
  sourceTaskId: 22,
  contractAddress: "0xb625B9...f5D676",
  targetAddress: "0x...",
  method: "stopAndWithdraw"  // or "pauseThenWithdraw"
}

ReturnService (return.service.ts):

1. initiateWithdraw(params):
   a. Get contract instance from deployed_contracts table
   b. Get wallet signer from WalletVault (sub-wallet that owns the contract)
   c. If method === "pauseThenWithdraw":
      - Call contract.pause(), wait confirmation
      - Verify contract.paused() === true on-chain
      - Wait 5 seconds for any pending batch txs
      - Call contract.withdrawRemaining(), wait confirmation
   d. If method === "stopAndWithdraw":
      - Call contract.stopAndWithdraw() — atomic single tx
      - Wait confirmation
   e. Verify token balance received by sub-wallet
   f. Record transaction
   g. Publish WITHDRAW_COMPLETED event

2. transferToReturnWallet(campaignId):
   a. Get sub-wallet balance (all tokens + ETH)
   b. Get target address from campaign config
   c. SECURITY CHECK — validateReturnAddress(targetAddress):
      - Is address in return_address_whitelist? 
      - If not: check return_address_changes table
      - If address changed < 24 hours ago: REJECT with time-lock error
        Return { error: "TIME_LOCK_ACTIVE", unlockAt: timestamp }
      - If address changed >= 24 hours ago: ALLOW
   d. Transfer all ERC-20 tokens to target
   e. Transfer remaining ETH (minus gas reserve of 0.005 ETH) to target
   f. Publish RETURN_COMPLETED event

3. Address whitelist management:
   POST /api/return-address/whitelist { address, label }
   GET /api/return-address/whitelist
   DELETE /api/return-address/whitelist/:address
   PUT /api/return-address/:campaignId { newAddress }
     - Requires re-authentication (fresh JWT < 5 min old)
     - Records change in return_address_changes with timestamp
     - Returns { newAddress, activeAfter: "2026-03-18T12:00:00Z" }
   GET /api/return-address/:campaignId/status
     - Returns { address, isWhitelisted, timeLockActive, unlockAt }

4. Database tables:
   return_address_whitelist: id, address, label, added_by, created_at
   return_address_changes: id, campaign_id, old_address, new_address, 
     changed_by, changed_at, active_after (changed_at + 24h)

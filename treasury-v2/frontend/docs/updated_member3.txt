 STEP 1: AUTH SYSTEM — Add login before existing modals
───────────────────────────────────────

@workspace The existing app has task creation modals (Create Swap Task, Create Task, Create Withdraw Task) with a dark theme UI but NO authentication. Anyone can access everything.

Add a complete auth system that gates ALL existing functionality:

1. Login Page (same dark theme as existing modals):
   - Dark background matching existing app (#1a1a2e or similar)
   - Centered card with:
     "Treasury Distributor" title
     Email input (dark bordered, same style as existing inputs)
     Password input (same style)
     "Sign In" button (blue, same as existing "Submit" buttons)
   - Error messages: "Invalid credentials" / "Account suspended"
   - After login: redirect to main dashboard

2. AuthService (auth.service.ts):
   - register(email, password, role):
     * Hash with bcryptjs (12 rounds)
     * Default role = VIEWER
     * Only ADMIN can create ADMIN/OPERATOR users
   - login(email, password):
     * Validate credentials
     * Rate limit: 5 failed attempts per 15 min (Redis counter)
     * 10 consecutive failures: suspend account
     * Return { accessToken (JWT, 15min), refreshToken (7 days) }
     * JWT payload: { userId, email, role, iat, exp }
   - refreshToken(token): Issue new access token
   - verifyToken(token): Decode + validate

3. RBAC Roles:
   ADMIN: Everything. Manage users, approve templates, change return addresses.
   OPERATOR: Create tasks, start/pause/stop airdrops, fund wallets, execute swaps.
   VIEWER: Read-only. View tasks, dashboards, logs.

4. Middleware:
   - authMiddleware: Check Bearer token on ALL /api/ routes
   - roleMiddleware(minRole): ADMIN > OPERATOR > VIEWER
   - freshAuthMiddleware: For return wallet changes, require JWT < 5 min old

5. Frontend auth context (auth-context.tsx):
   - AuthProvider wrapping the app
   - Store tokens in memory (NOT localStorage — security)
   - Auto-refresh before expiry
   - useAuth() hook: { user, login, logout, isAuthenticated, role }

6. Gate existing modals:
   - Create Swap Task: requires OPERATOR or ADMIN
   - Create Task: requires OPERATOR or ADMIN  
   - Create Withdraw Task: requires OPERATOR or ADMIN
   - View tasks: requires any authenticated user
   - Admin functions: requires ADMIN only

7. Add user info to top-right of existing UI:
   Small area showing: "admin@treasury.local" + role badge + "Logout" link
   Same dark theme styling.

API endpoints:
  POST /api/auth/login { email, password }
  POST /api/auth/register { email, password, role } — admin only
  POST /api/auth/refresh { refreshToken }
  POST /api/auth/logout

Database: users table, sessions table

 STEP 4: CAMPAIGN ORCHESTRATOR (backend)
───────────────────────────────────────

@workspace Create the campaign orchestrator service. This is the backend brain that coordinates between wallet-service and contract-service.

CampaignOrchestratorService (campaign-orchestrator.service.ts):

This service does NOT interact with blockchain directly. It calls:
- wallet-service (port 3001) for wallet + swap operations
- contract-service (port 3002) for deploy + airdrop operations

STATE MACHINE — enforce these transitions:
CREATED → WALLET_GENERATED → FUNDED → SWAPPING → SWAPPED → 
DEPLOYING → DEPLOYED → CONTRACT_FUNDED → AIRDROP_ACTIVE →
PAUSED ↔ AIRDROP_ACTIVE → STOPPED → WITHDRAWING → WITHDRAWN → 
RETURNED → COMPLETED

Error states: SWAP_FAILED (retry from FUNDED), DEPLOY_FAILED (retry from SWAPPED)

Every action:
1. Verify current state allows the transition
2. Call the appropriate service API
3. Record state change in campaign_state_history table
4. Emit event via Socket.io for live UI updates

Methods:
- createCampaign(config) → save CREATED, call wallet-service to generate sub-wallet
- fundCampaign(id) → call wallet-service to fund sub-wallet  
- startSwap(id, swapConfig) → call wallet-service to execute swap
- deployContract(id, templateId) → call contract-service to deploy
- fundContract(id) → call contract-service to fund contract
- startAirdrop(id, recipients) → call contract-service to start airdrop
- pauseAirdrop(id) → call contract-service
- resumeAirdrop(id) → call contract-service  
- stopAirdrop(id) → call contract-service
- withdrawRemaining(id) → call contract-service
- transferToReturn(id) → call wallet-service

Event Consumer (events/consumer.ts):
- Subscribe to RabbitMQ exchange "treasury.events"
- For each event: update campaign state + emit Socket.io update + save log
- Events: WALLET_GENERATED, WALLET_FUNDED, SWAP_COMPLETED, SWAP_FAILED, 
  CONTRACT_DEPLOYED, CONTRACT_FUNDED, AIRDROP_STARTED, AIRDROP_PAUSED,
  AIRDROP_STOPPED, WITHDRAW_COMPLETED, RETURN_COMPLETED

API Client (lib/api-client.ts):
- WalletServiceClient: typed methods for http://wallet-service:3001/api/*
- ContractServiceClient: typed methods for http://contract-service:3002/api/*
- Both: forward JWT in Authorization header, 30s timeout, retry on 5xx

Campaign database:
campaigns: id, name, status (all states), user_id, sub_wallet_address,
  contract_addresses (json array), return_wallet_address, token_address,
  target_token_address, fund_amount, transfer_limit, gas_threshold,
  transfer_count, total_distributed, batch_size, recipient_count,
  created_at, updated_at, completed_at

campaign_state_history: id, campaign_id, from_state, to_state,
  triggered_by (userId or "SYSTEM"), reason, tx_hash, timestamp

 STEP 5: REAL-TIME + SOCKET.IO
───────────────────────────────────────

@workspace Set up Socket.io for real-time dashboard updates. The task detail page and main dashboard need live data without polling.

NotificationService (notification.service.ts):
1. Socket.io server on port 3000, namespace /live
2. Authentication: validate JWT from handshake auth header
3. Rooms:
   - "dashboard" — all connected users (for stat card updates)
   - "task:{taskId}" — users viewing a specific task detail
4. Events emitted to clients:
   - "stats:update" { activeTasks, totalContracts, totalTransfers, gasPrice }
   - "task:update" { taskId, status, transferCount, batchesCompleted, ... }
   - "task:batch" { taskId, batchId, status, txHash, gasUsed }
   - "task:gas" { taskId, gasPrice, isAcceptable, threshold }
   - "task:error" { taskId, message, severity }
   - "alert" { message, severity, taskId? }

Frontend Socket.io (lib/socket.ts):
- useTaskSocket(taskId) hook:
  * Connect to /live namespace with JWT
  * Join "task:{taskId}" room
  * Return { stats, batches, events, isConnected }
  * Auto-reconnect on disconnect
  
- useDashboardSocket() hook:
  * Join "dashboard" room
  * Return { globalStats, recentEvents }

RabbitMQ Consumer → Socket.io bridge:
- For each RabbitMQ event from wallet-service/contract-service:
  a. Save to event_logs (append-only)
  b. Update campaign state
  c. Emit appropriate Socket.io event to correct rooms

 STEP 7: LOGGING & AUDIT TRAIL
───────────────────────────────────────

@workspace Create the logging system and log viewer page. Dark theme.

LogService (log.service.ts):
1. logEvent(event):
   - Generate ULID (time-ordered ID)
   - Calculate chain hash: sha256(JSON.stringify(event) + previousHash)
   - INSERT into event_logs — NEVER update or delete
   
2. queryLogs(filters):
   - Filters: taskId, eventType, severity, dateFrom, dateTo, actor
   - Pagination: page, limit (max 100)
   - Return { logs, total, pages }

3. exportLogs(filters, format): CSV or JSON stream

4. logAuditAction(userId, action, resource, oldValue, newValue, req):
   - Save to audit_trail table with IP + user agent

Database:
event_logs: id (ULID), task_id, event_type, severity (INFO|WARN|ERROR|CRITICAL),
  actor_type (USER|SYSTEM), actor_id, payload_json, tx_hash, chain_hash, timestamp
  NO UPDATE/DELETE permissions on this table.

audit_trail: id, user_id, action, resource_type, resource_id,
  old_value_json, new_value_json, ip_address, user_agent, timestamp

Log Viewer Page (/logs) — dark theme:
- Filter bar at top:
  * Task dropdown (multi-select)
  * Event Type dropdown (multi-select): FUND, SWAP, DEPLOY, AIRDROP, WITHDRAW, ERROR
  * Severity pills: [INFO] [WARN] [ERROR] [CRITICAL] — click to toggle
  * Date range picker
  * "Export CSV" button / "Export JSON" button
- Log table:
  | Time | Task | Event | Severity (badge) | Actor | Details | Tx Hash |
  Severity badges: INFO=gray, WARN=amber, ERROR=red, CRITICAL=red-bold
  "Details" column is expandable — click to show full payload JSON
  Tx Hash links to Etherscan
- Pagination at bottom: 50 rows per page
